## `spec.md`

```markdown
# Chi Shell — Quick Settings & Status Bar Specification

> Version 0.3.0-draft
> Target: Wayfire compositor, Wayland, Fedora Linux
> Stack: Qt 6 / QML, LayerShellQt, C++17

---

## 1. Overview

The Quick Settings panel is Chi Shell's primary control surface — a single
overlay that gives the user instant access to the most common system controls
without ever opening a full Settings application.

**Design philosophy:**
- Android-style tile grid for tap-to-toggle / tap-to-expand interactions
- macOS-style polish: smooth animations, resizable tiles, media integration
- freedesktop-native: every backend is a standard D-Bus interface, no hacks
- Productivity-first: one-click access, keyboard shortcuts, minimal latency

The Status Bar is the persistent top strip that displays real-time system
state (clock, battery, network, tray icons) and provides the entry point to
both Quick Settings and the Notification Center.

---

## 2. Freedesktop & Linux Standards Used

Every feature maps to an established D-Bus interface or freedesktop spec.
Chi Shell acts as a **host/client** for all of these — it never invents a
proprietary protocol when a standard exists.

| Feature              | Standard / D-Bus Interface                                  | Bus     |
|----------------------|-------------------------------------------------------------|---------|
| Battery              | `org.freedesktop.UPower` (DisplayDevice)                    | System  |
| Network (Wi-Fi)      | `org.freedesktop.NetworkManager`                            | System  |
| Bluetooth            | `org.bluez.Adapter1`, `org.bluez.Device1`                   | System  |
| Audio volume/mute    | PipeWire/PulseAudio via `pactl` or native PipeWire D-Bus    | Session |
| Brightness           | `/sys/class/backlight` (sysfs) + `brightnessctl` fallback   | N/A     |
| Media playback       | `org.mpris.MediaPlayer2` (MPRIS v2.2)                       | Session |
| System tray icons    | `org.kde.StatusNotifierItem` / `StatusNotifierWatcher` (SNI)| Session |
| Notifications        | `org.freedesktop.Notifications` (fd.o Notifications v1.2)   | Session |
| Power profiles       | `net.hadess.PowerProfiles` (power-profiles-daemon)          | System  |
| App actions          | `Desktop Entry Spec` — `Actions=` key (Quicklists)          | N/A     |
| Do Not Disturb       | Managed internally (inhibit fd.o Notifications delivery)     | Session |
| Night Light          | `org.gnome.SettingsDaemon.Color` / gammastep D-Bus or manual| Session |
| Airplane Mode        | `org.freedesktop.NetworkManager` (WirelessEnabled=false) +  |         |
|                      | `org.bluez` (Powered=false) combined toggle                 | System  |
| Screen lock          | `org.freedesktop.login1.Session.Lock()`                     | System  |
| Session (logout/etc) | `org.freedesktop.login1.Manager`                            | System  |

## 3. Status Bar (`statusbar/`)

The Status Bar is a 36px-tall `LayerShellQt` surface anchored to the top
edge with exclusive zone, always visible.

### 3.1 Layout

```
┌─────────────────────────────────────────────────────────────────┐
│ [clock]              [tray icons . . .]    [net] [bt] [bat 82%]│
│  10:42 AM                                    ▼ click → QS      │
└─────────────────────────────────────────────────────────────────┘
```

- **Left region:** Clock (time, optionally date on hover/click)
  - Click → open Notification Center
- **Center region:** System tray icons (SNI items)
- **Right region:** Network, Bluetooth, Battery status pill
  - Click → open Quick Settings panel

### 3.2 System Tray (SNI Host)

Chi Shell implements a **StatusNotifierHost** and a **StatusNotifierWatcher**
on the session bus. This is the standard mechanism for apps to expose tray
icons on modern Linux.

**Implementation:**
- Register as `org.kde.StatusNotifierWatcher` on the session bus
- Listen for `RegisterStatusNotifierItem` calls from apps
- For each registered item, read properties from the
  `org.kde.StatusNotifierItem` interface:
  - `IconName` / `IconPixmap` — the icon to display
  - `Title` — tooltip text
  - `Status` — `Active`, `Passive`, or `NeedsAttention`
  - `Menu` — `com.canonical.dbusmenu` object path for the right-click menu
  - `ItemIsMenu` — if true, left-click opens menu instead of Activate
- Render each item as a small icon in the status bar center region
- Left-click → call `Activate(x, y)` on the item (or open menu if ItemIsMenu)
- Right-click → render the `com.canonical.dbusmenu` tree as a popup menu

**Signals to watch:**
- `NewIcon`, `NewTitle`, `NewStatus` → update rendering
- `StatusNotifierItemRegistered` / `Unregistered` on the watcher

**Fallback:** If no SNI watcher is running, apps fall back to XEmbed system
tray. Chi Shell is Wayland-only so we do NOT implement XEmbed. Apps that
only support XEmbed will not get tray icons — this is standard Wayland
behavior (same as GNOME, KDE Plasma on Wayland).

### 3.3 Clock

- Displays time in user's locale format (12h/24h from system locale)
- Updated every 30 seconds (not every 1s — saves CPU, minute precision)
- On hover: show full date tooltip
- On click: toggle Notification Center

### 3.4 Status Icons (Right Pill)

Small icons showing at-a-glance state:
- Network: signal strength icon (from `networkIcon` in SystemStatus)
- Bluetooth: on/off icon
- Battery: level + charging indicator, percentage text
- Optional: DND indicator (small moon icon when active)

### 4.3 Section A — Status Header

- Horizontal slider bar with filled portion in `primaryContainer` color

- **Optimization:** Same debounce strategy as brightness
- **Scroll wheel:** If the panel is open and the mouse is over the volume

Displays the currently active media player, if any. Hidden when no media

**Backend: MPRIS v2.2**

- Use `playerctld` logic: track the most recently active player
- Read properties from `org.mpris.MediaPlayer2.Player`:
  - `PlaybackStatus` — Playing, Paused, Stopped
  - `Metadata` — `xesam:title`, `xesam:artist`, `mpris:artUrl`
  - `CanGoNext`, `CanGoPrevious`, `CanPlay`, `CanPause`
- Subscribe to `PropertiesChanged` signal for live updates

## 7. Performance Optimizations

### 7.1 Polling Strategy (SystemStatus)

| Data source        | Method          | Poll interval  | Notes                              |
|--------------------|-----------------|----------------|------------------------------------|
| Battery            | D-Bus property  | 15s            | UPower, always fast                |
| Network state      | D-Bus property  | 15s            | NM State property, always fast     |
| Wi-Fi SSID/signal  | `nmcli` async   | 15s            | With `--wait 0`, backoff on fail   |
| Bluetooth          | D-Bus property  | 15s            | BlueZ adapter property             |
| Audio volume       | `pactl` async   | 15s            | With 2s timeout, backoff on fail   |
| Audio mute         | `pactl` async   | 15s            | Same as above                      |
| Brightness         | sysfs read      | 15s            | File I/O, always fast              |
| MPRIS metadata     | D-Bus signals   | Event-driven   | Only active when QS panel is open  |
| MPRIS position     | D-Bus property  | 1s (when open) | Only when Playing & card visible   |
| SNI tray items     | D-Bus signals   | Event-driven   | Signal-driven, no polling          |
| Power profile      | D-Bus property  | 30s            | Rarely changes                     |

### 7.2 Async Command Backoff (existing, keep it)

The current `asyncCommand()` system with `shouldSkip()` / `recordSuccess()` /
`recordFailure()` is solid:
- 0–2 failures: always retry
- 3–9 failures: try once every 4 poll cycles (~60s)
- 10+ failures: stop entirely
- 2-second timeout per process
- Duplicate command prevention via `m_runningCommands` set

### 7.3 Slider Debouncing

Current code calls `setBrightness()` / `setVolume()` on every
`onPositionChanged` pixel. This spawns a `brightnessctl` / `pactl` process
on every mouse move event.

**Fix:** Add a 50ms debounce timer:
```qml
Timer {
    id: brightnessDebounce
    interval: 50
    onTriggered: systemStatus.brightness = pendingBrightness
}
// In MouseArea:
onPositionChanged: {
    pendingBrightness = Math.max(0, Math.min(1, mouse.x / width))
    brightnessDebounce.restart()
}
```

### 7.4 Signal-Driven Updates (replace polling where possible)

For services that support D-Bus signals, connect to `PropertiesChanged`
instead of polling:
- **UPower:** `org.freedesktop.DBus.Properties.PropertiesChanged` on the
  DisplayDevice object → update battery immediately on plug/unplug
- **NetworkManager:** `StateChanged` signal → update network immediately
- **BlueZ:** `PropertiesChanged` on adapter → update BT status immediately
- **Power Profiles:** `PropertiesChanged` on `net.hadess.PowerProfiles`

Keep the 15s poll as a **fallback heartbeat** to catch anything signals miss,
but the UI should feel instant thanks to signals.

### 7.5 Lazy Loading

- Media card: QML `Loader` with `active: mprisService.hasActivePlayer`
- Sub-panels (WifiSheet, BluetoothSheet): `Loader` with `active: false`,
  set `active: true` only when the user taps the expand arrow
- Tray icon menus: only fetch `com.canonical.dbusmenu` tree on right-click,
  not on registration

### 7.6 Resource Cleanup on Panel Hide

When `shell.quickSettingsOpen` becomes `false`:
- Disconnect MPRIS `PropertiesChanged` watchers
- Stop MPRIS position polling timer
- Stop any Bluetooth discovery scan
- Cancel pending Wi-Fi scan

## Key Takeaways from the Research

1. The StatusNotifierItem specification "defines the management of visual items, usually icons used for reporting the status of an application to the user or provide quick access to common actions performed by that application." This is the **only** proper way to do tray icons on Wayland — XEmbed doesn't work.

2. In reality, "Freedesktop is more of a loose collection of specifications," and the StatusNotifierItem spec "provides a way for applications to create and manage a 'system tray' icon." DEs like KDE have integrated first-party support (KStatusNotifierItem) for the specification. You'll want to implement both the **Watcher** and the **Host** side.

3. MPRIS (Media Player Remote Interfacing Specification) is a standard D-Bus interface which aims to provide a common programmatic API for controlling media players. It provides a mechanism for discovery, querying and basic playback control of compliant media players, as well as a tracklist interface. This is how your media card will work — purely signal-driven, no polling needed for metadata.

4. The Power Profile Daemon offers a standardized way to control system power profiles through a DBus interface, providing three main profiles (power-saver, balanced, and performance). It's a "freedesktop.org-compliant service that centralizes power profile management across Linux distributions."

5. There will be a single `org.freedesktop.StatusNotifierWatcher` service instance registered on the session bus. The StatusNotifierWatcher service is used to keep track of StatusNotifierItem instances, enumerate them and notify when new ones are registered or old ones are unregistered. It is also used to keep track of StatusNotifierHost instances.

There is **no** single "quick controls API" standard on freedesktop — instead, each system service (NetworkManager, BlueZ, UPower, power-profiles-daemon, MPRIS) exposes its own D-Bus interface, and the shell acts as the **unified frontend** that stitches them all together. That's exactly what you're building. 🔥
